CalEnviroScreen Environmental Justice Explorer work in progress

An interactive map and AI assistant for exploring pollution burden, social vulnerability, and disadvantaged-community designations across California — and the places where conservation investment overlaps with environmentally burdened communities.

← Open the App

About this project

A research collaboration at the Department of Environmental Science, Policy, and Management at UC Berkeley, in partnership with the Trust for Public Land and with support from NASA.

This app helps California environmental-justice advocates, community organizations, and policymakers explore where pollution burden falls most heavily, which communities are formally designated as disadvantaged, and how conservation investment intersects with environmentally burdened places. The interface lets you toggle 16 geospatial layers and ask plain-English questions that translate into live SQL queries against a federated catalog of cloud-native datasets.

This is a work in progress built on draft CalEnviroScreen 5.0 (released January 2026 by OEHHA). Final scores from OEHHA may shift during finalization.

How it works

The map renders open geospatial datasets directly from cloud-native formats — vector tiles (PMTiles), Cloud-Optimized GeoTIFFs, and GeoParquet — hosted on the National Research Platform. No tile server is involved; your browser fetches only the bytes needed for the current view.

The chat assistant translates plain-English questions into DuckDB SQL over the same Parquet files. Dataset descriptions and column schemas are discovered automatically through a SpatioTemporal Asset Catalog (STAC), so the assistant sees the same column definitions you do.

Open models on shared infrastructure. The assistant is powered by open-weight large language models (GLM-4.7, Qwen3, GPT-OSS 120B, Kimi K2.5, MiniMax M2.5) running on the NRP open-llm-proxy, a university inference service on hardware powered substantially by renewable energy. Compared with large commercial inference services, this approach reduces both the energy and the data-privacy footprint of every conversation. Carbon and energy use are tracked on our companion sustainability dashboard.

The full agent stack — map, chat, MCP tool server, and STAC catalog — is open-source. See the project repository and the geo-agent core library.

Transparent, reproducible queries

Every chat answer shows the SQL the assistant ran. The same query can be copied into DuckDB, the DuckDB web shell, R, Python, or any tool that reads Parquet over HTTPS — no API keys, no signup, no rate limits. The Parquet files are public.

For example, an assistant answer to "Which census tracts in Alameda County are in the top 10% of CES burden, and which indicators drive the score?" might run:

INSTALL httpfs; LOAD httpfs;

SELECT tract, county, AppoxLoc,
       CIscore, CIscore_Pctl,
       AirPM25_Pctl, DieselPM_Pctl, DrinkingWater_Pctl,
       Asthma_Pctl, Poverty_Pctl
FROM read_parquet(
    'https://s3-west.nrp-nautilus.io/public-calenviroscreen/calenviroscreen-5-0/ces5.parquet'
)
WHERE county = 'Alameda'
  AND CIscore_Pctl >= 90
ORDER BY CIscore_Pctl DESC;

Paste that query into DuckDB and you get the same answer, with the same provenance, today and three years from now. No screenshot, no PDF export — just the question, the SQL, and the data.

Expanding queries

The assistant's queries are starting points, not endpoints. Three patterns that work well:

1. Change the filter

The first query found the top 10% of tracts in Alameda County. Swap county = 'Alameda' for a different county, or drop the county filter and add AND Pop23 > 5000 to surface populous burdened tracts statewide. Every column and threshold is documented in the CES data dictionary.

2. Join layers spatially

Each layer is a separate Parquet. DuckDB's spatial extension joins them in-place — no download, no PostGIS:

INSTALL spatial; LOAD spatial;
INSTALL httpfs;  LOAD httpfs;

-- Which Assembly districts contain the most high-CES population?
SELECT a.NAMELSAD AS district,
       SUM(c.Population) FILTER (WHERE c.CIscore_Pctl >= 90) AS high_burden_pop,
       SUM(c.Population) AS total_pop
FROM read_parquet('https://s3-west.nrp-nautilus.io/public-calenviroscreen/calenviroscreen-5-0/ces5.parquet') c
JOIN read_parquet('https://s3-west.nrp-nautilus.io/public-census/census-2025/sldl/sldl.parquet') a
  ON ST_Intersects(c.geometry, a.geometry)
WHERE a.STATEFP = '06'
GROUP BY a.NAMELSAD
ORDER BY high_burden_pop DESC NULLS LAST
LIMIT 20;

3. Switch to the H3 hex tiles for fast aggregation

Most datasets are also published as H3 hexagon Parquet files partitioned by resolution-0 cell. Joins on hex ID are 10–100× faster than geometric joins for nationwide aggregation. Each STAC collection lists its hex asset alongside the polygon Parquet.

Tip: Ask the assistant to "rewrite that query using the H3 hex tiles," or to "show me the same thing for Imperial County." It will adapt the SQL — and you can always paste the final version into your own DuckDB session for further exploration.
